# How to Read a GitHub Work Trail

A GitHub work trail is a map, not a verdict.

It helps future readers understand what happened across repositories, pull requests, commits, and receipts without inheriting authority from the author.

## 1. Start with the Window

Read the date range first. The window tells you what period the trail covers.

## 2. Identify the Repositories Observed

Look at the repositories listed in the trail. Each repository may represent a different lane: public proof, validation machinery, canon surface, governance surface, or replay surface.

Do not assume all repositories have the same evidentiary weight.

## 3. Walk the Pull Requests

Pull requests show intent, review surface, and proposed structure. A pull request may show work, but it is not proof by itself.

Treat PR titles and bodies as observed metadata.

## 4. Walk the Commits

Commits provide stronger sequence evidence than summaries.

Check commit SHA, timestamp, repository, message, linked files, and parent chain when needed.

A commit is a pointer in the chain, not automatic proof of correctness.

## 5. Treat PASS, GREEN, and MERGED Carefully

Words like PASS, GREEN, VERIFIED, and MERGED must be read as observed language unless independently replayed.

A trail may record those words without endorsing them.

## 6. Follow the Reader Path

The follow_trail section gives a recommended reading order.

Use it to walk the work forward and backward. It reduces confusion. It does not create authority.

## 7. Verify Evidence Separately

A clean trail keeps explanation separate from proof.

To verify a claim, independently inspect or replay the underlying evidence: files, hashes, workflows, commits, PR diffs, and external witnesses.

## Core Rule

Do not turn breadcrumbs into a crown.

A trail helps you find the evidence. It does not become the evidence.
