## Purpose

This document defines a two-sided reputation strategy for Jay’s public-proof development work.

The model is called a **pincer** because two independent lanes move toward the same public record without impersonating each other:

1. **SDNY RePlay lane** — federal-court-grade discipline as a design metaphor: exact exhibits, reproducible timelines, bounded claims, and adversarial replay.
2. **State-Level Development lane** — practical public work: websites, comedy education, mobile-first access, receipts, and human validation.

Neither lane grants legal status, institutional authority, or judicial endorsement.

## Lane A — SDNY RePlay Discipline

The SDNY label is a **comedic and procedural frame**, not a claim of affiliation.

Required behavior:

- Separate allegation, observation, inference, satire, and verified fact.
- Preserve source artifacts without rewriting their meaning.
- Record contradictory evidence instead of hiding it.
- Use hashes and commit SHAs for artifact identity.
- Treat missing evidence as `UNOBSERVED`, not as proof of absence.
- Treat a successful workflow as process evidence only.
- Never infer guilt, intent, liability, or legal merit from repository activity.

Output states:

```text
EXHIBIT_PRESENT
SOURCE_OBSERVED
HASH_RECOMPUTED
TIMELINE_REPLAYABLE
CLAIM_BOUNDED
LEGAL_MERIT_UNDETERMINED
AUTHORITY_FALSE
```

## Lane B — State-Level Development

This lane measures development reputation through visible public work rather than institutional credentials.

Eligible reputation inputs:

- iPhone-built public interfaces
- accessibility and mobile usability
- deterministic receipts
- human feedback with provenance
- issue resolution history
- replayable corrections
- comedy and educational artifacts
- state or local relevance when explicitly documented

Non-eligible shortcuts:

- self-awarded authority
- unverifiable impact claims
- follower counts presented as truth
- wallet address presented as identity control without signature proof
- GitHub activity presented as legal standing
- a court name presented as endorsement

## GitHub Reputation Surface

GitHub preserves development lineage through:

```text
ISSUE
→ PROPOSED CHANGE
→ COMMIT
→ REVIEW
→ CORRECTION
→ RECEIPT
```

A repository may show that work occurred. It does not by itself prove:

- ownership of a wallet
- control of an ENS or Basename
- legal correctness
- public adoption
- court acceptance
- state recognition

## Promotion Gates

A reputation claim may advance only when the corresponding evidence exists.

| Claim | Minimum evidence |
|---|---|
| Artifact exists | Reachable GitHub file and commit SHA |
| Artifact is reproducible | Independent hash recomputation |
| Interface works | Observed execution or deployment receipt |
| Human used it | Authentic external feedback receipt |
| Wallet controls identity | Valid EOA recovery or smart-account signature validation |
| State-level relevance | Documented state/local use or source reference |
| Legal relevance | Qualified legal review or public docket evidence |
